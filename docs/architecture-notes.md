# Direct Ride Architecture Notes

These notes capture the reasoning, tradeoffs, and follow-up work behind the Direct Ride Terraform stack. The README explains what the repository deploys and how to run it; this document explains why the infrastructure is shaped this way and what to consider as the project grows.

## Current Environment

The repository currently defines a `dev` environment in `environments/dev`. It is designed to be practical for early application development while still using production-like AWS building blocks:

- VPC networking with public, private application, and private database tiers
- ECS Fargate for the backend API
- Application Load Balancer for public API ingress
- RDS PostgreSQL for application data
- S3 website hosting for the frontend
- ECR for backend container images
- Secrets Manager and SSM Parameter Store for runtime configuration
- IAM roles for ECS and optional GitHub Actions deployment

The environment favors clear separation and low operational overhead over full production redundancy.

## Traffic Flow

Frontend traffic:

1. Static frontend files are uploaded to the S3 website bucket.
2. Users access the frontend through the S3 website endpoint.
3. The frontend calls the backend API through the backend Application Load Balancer URL.

Backend API traffic:

1. Public HTTP traffic reaches the Application Load Balancer in public subnets.
2. The ALB forwards requests to ECS Fargate tasks in private application subnets.
3. ECS tasks connect to the PostgreSQL RDS instance in private database subnets.
4. ECS tasks read secrets from Secrets Manager and non-secret config from SSM Parameter Store.

Deployment traffic:

1. A GitHub Actions workflow can assume the deploy role through GitHub OIDC when `github_repository` is configured.
2. The deploy role can push backend images to ECR, register task definitions, update the ECS service, upload frontend files to S3, and invalidate CloudFront distributions when distribution ARNs are supplied.

## Networking Decisions

The VPC uses three subnet tiers across two Availability Zones:

- Public subnets host internet-facing infrastructure such as the ALB and NAT gateway.
- Private app subnets host ECS Fargate tasks.
- Private database subnets host RDS and do not have a default route to the internet.

The dev environment uses a single NAT gateway in the first public subnet. This is cost-conscious and acceptable for development, but it creates an Availability Zone dependency for outbound traffic from private app subnets. A production environment should usually use one NAT gateway per Availability Zone.

The database subnets intentionally do not route through the NAT gateway. This keeps the database tier isolated and limits outbound paths from the data layer.

## Security Model

The current security posture is:

- The ALB is public.
- ECS tasks are private and only receive application traffic from the ALB security group.
- RDS is private and only allows PostgreSQL traffic from the ECS task security group.
- The backend database password is generated and managed by RDS in Secrets Manager.
- The JWT signing secret is generated by Terraform and stored in Secrets Manager.
- Non-secret application configuration can be stored under an SSM Parameter Store path.
- The ECS task execution role can read secrets needed during task startup.
- The ECS application task role can read configured application secrets and SSM parameters.

GitHub Actions OIDC is optional. When enabled, it avoids long-lived AWS access keys in GitHub by allowing a specific repository and branch to assume a deploy role.

## Application Runtime Notes

The backend API is deployed as an ECS Fargate service. The task receives:

- `DB_HOST`
- `DB_NAME`
- `DB_PORT`
- `DB_USER`
- `DB_PASSWORD`
- `JWT_SECRET`

`DB_PASSWORD` and `JWT_SECRET` are injected as ECS secrets, not plain environment variables.

The ECS service uses a target group health check. The default health check path is `/health`, so the backend container should expose that route and return a `200-399` response when healthy.

The default container image tag is `latest`. That is convenient for early development, but immutable version tags are safer for repeatable deployments.

## Frontend Hosting Notes

The frontend is currently hosted directly from an S3 website bucket. This keeps the setup simple, but it has some limitations:

- S3 website endpoints are HTTP-only.
- The bucket policy allows public object reads.
- There is no CDN, custom domain, TLS certificate, or edge caching resource in this Terraform stack yet.

For production, CloudFront should sit in front of S3 with an ACM certificate and a tighter bucket access pattern.

## API Ingress Notes

The compute module currently creates an HTTP listener on port `80`. The ALB security group also allows HTTPS ingress on port `443`, but the stack does not yet create:

- An ACM certificate
- An HTTPS listener
- HTTP-to-HTTPS redirect behavior
- Domain records

Those should be added before treating the API endpoint as production-ready.

## Data Protection Notes

The RDS instance uses encrypted `gp3` storage and AWS-managed master password storage. Dev defaults make teardown easy:

- `db_deletion_protection = false`
- `db_skip_final_snapshot = true`

Those defaults are suitable for disposable dev data. Production should use stricter settings:

- Enable deletion protection.
- Require a final snapshot.
- Increase backup retention.
- Consider point-in-time recovery expectations.
- Define maintenance and backup windows.
- Consider read replicas or higher availability options if required.

## Operational Runbooks

### Deploy a New Backend Image

1. Build and push a backend image to the ECR repository output by Terraform.
2. Set or update `backend_api_image_tag`.
3. Run `terraform plan` from `environments/dev`.
4. Run `terraform apply`.

If using GitHub Actions, the deploy workflow can push the image, register a new task definition, and update the ECS service through the OIDC deploy role.

### Check Backend Logs

Backend logs are written to the CloudWatch log group:

```text
/ecs/<project_name>-<environment>/api
```

For the default dev environment:

```text
/ecs/direct-ride-dev/api
```

### Investigate Failed Health Checks

Check these first:

- The container is listening on `backend_api_container_port`.
- The app returns `200-399` on `backend_api_health_check_path`.
- The ECS task can start with the configured CPU and memory.
- Required secrets and environment variables are present.
- The ECS task security group allows traffic from the ALB security group.
- The target group shows registered targets.

### Rotate JWT Secret

The JWT secret is generated by the `random_password` resource and stored in Secrets Manager. Rotating it will invalidate tokens signed with the old value. Plan the rotation around user sessions and application behavior.

### Rotate Database Password

The RDS master password is managed by AWS through Secrets Manager. If application-level users are introduced later, prefer rotating those credentials separately from the master user and avoid using the master user for normal application traffic.

## Cost Considerations

The main always-on cost drivers in dev are likely to be:

- NAT gateway
- RDS instance and storage
- Application Load Balancer
- ECS Fargate tasks
- CloudWatch logs

To reduce dev cost, consider lowering ECS desired count when not in use, right-sizing the database, setting log retention thoughtfully, and evaluating whether NAT gateway availability is needed for every dev scenario.

## Production Hardening Backlog

Before introducing a production environment, consider adding:

- Separate `prod` environment composition
- Remote Terraform state with locking
- CloudFront distribution for the frontend
- ACM certificates and custom domains
- HTTPS listener and HTTP-to-HTTPS redirects for the API
- Route 53 DNS records
- One NAT gateway per Availability Zone
- ECS service autoscaling
- RDS deletion protection and stricter snapshot behavior
- Database backup and maintenance windows
- CloudWatch alarms for ALB, ECS, RDS, and NAT gateway health
- WAF for public endpoints
- VPC endpoints for AWS services used by ECS tasks
- Immutable backend image tags in deployment workflows
- Least-privilege review of deploy and application IAM policies
- Dedicated application database user instead of relying on the master user

## Open Questions

These are useful decisions to make as the application matures:

- Will the frontend be served through CloudFront with a custom domain?
- Will the API use the same domain under a path, or a separate subdomain?
- Should backend deployments be driven entirely by GitHub Actions?
- What image tagging strategy should be used for release promotion?
- What is the expected production RDS availability and backup target?
- Does the application need an uploads bucket managed by this stack?
- Does email delivery through SES belong in the core stack or a separate module?
- Should app config remain in SSM Parameter Store, or move into a higher-level config management approach?
